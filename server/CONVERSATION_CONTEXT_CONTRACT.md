# Conversation Context Contract

## 1. Scope

This document is the normative SV-US-010 contract for durable conversation identity, bounded
history, provider-neutral turn audit records, PostgreSQL graph checkpoints, deletion, anonymization,
and retention. SV-US-010 adds the authenticated history route and internal application boundaries;
SV-US-011 owns `POST /chat`, provider adapters, and the LangGraph workflow that consumes them.

Application audit rows and graph checkpoints are separate durability surfaces. Audit rows explain
what the product did; checkpoints support graph replay/recovery and are never the only conversation
record.

## 2. Visitor Identity And Ownership

Every conversation operation carries exactly one visitor identity:

- logged-in: `external_user_id`, trimmed public text matching
  `[A-Za-z0-9][A-Za-z0-9._:@-]{0,127}`;
- anonymous: `anonymous_session_id`, an opaque WordPress-generated value of 32 through 128
  characters matching `[A-Za-z0-9][A-Za-z0-9_-]*`.

Supplying neither or both is invalid. A logged-in request does not silently claim an anonymous
conversation and an anonymous request does not attach to a logged-in conversation. Account/session
linking is a future explicit workflow, not an ownership fallback.

The installation credential authenticates the WordPress server but does not confer ownership of
every conversation. Ownership is an exact match on the stored identity kind and value. For turn
continuation the conversation must also be `active`. History retrieval may return an owned `active`
or `archived` conversation. Missing, mismatched, deleted, malformed, and wrong-status lookups use the
same `404 conversation_not_found` response and never reveal whether an ID exists.

Conversation and graph thread IDs are application UUIDs generated by the server. A new conversation
stores `langgraph_thread_id=thread_<uuid>`; callers never choose or receive authority over an
arbitrary thread ID. `channel` is exactly `web`, `mobile`, or `admin_test` and cannot select a model
or provider.

## 3. Begin-Turn And Context Boundary

The internal `beginTurn` input contains only optional `conversation_id`, the exact-one visitor
identity, `channel`, and a validated application `correlation_id`. In one transaction it:

1. creates a durable conversation and unique graph thread when `conversation_id` is absent, or locks
   and verifies the owned active conversation when it is present;
2. creates one `conversation_turns` row with `status=processing` and a unique
   `(conversation_id, correlation_id)` boundary;
3. loads the stored summary and newest bounded messages, returning messages in chronological order;
4. returns the server-owned conversation ID, thread ID, turn ID, summary, recent messages, and
   `history_truncated` flag.

`CONVERSATION_HISTORY_MESSAGE_LIMIT` defaults to `20` and accepts `1..100`.
`CONVERSATION_SUMMARY_MAX_CHARS` defaults to `4000` and accepts `100..20000`. The summary is
provider-neutral application text. A summary update longer than the configured bound is rejected;
it is never silently truncated. Recent messages include `user`, `assistant`, `system`, and `tool`
roles, citations, safe metadata, and creation time. The query selects newest `limit + 1`, reports
truncation, retains only `limit`, then returns them oldest-first.

Retrying `beginTurn` with the same owned conversation and correlation ID returns the existing
processing turn and the same context rather than creating a duplicate. A completed correlation ID
cannot be started again.

## 4. Provider-Neutral Turn Audit

SV-US-010 adds a first-class turn table and links messages, tool calls, and usage to it:

```sql
ALTER TABLE conversations
  ADD COLUMN deleted_at TIMESTAMPTZ NULL,
  ADD COLUMN anonymized_at TIMESTAMPTZ NULL;

CREATE TABLE conversation_turns (
  id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  conversation_id UUID NOT NULL REFERENCES conversations(id) ON DELETE CASCADE,
  correlation_id TEXT NOT NULL,
  status TEXT NOT NULL DEFAULT 'processing'
    CHECK (status IN ('processing', 'succeeded', 'failed')),
  latency_ms INTEGER NULL CHECK (latency_ms IS NULL OR latency_ms >= 0),
  error_code TEXT NULL,
  metadata JSONB NOT NULL DEFAULT '{}'::jsonb,
  started_at TIMESTAMPTZ NOT NULL DEFAULT NOW(),
  completed_at TIMESTAMPTZ NULL,
  UNIQUE (conversation_id, correlation_id),
  CHECK ((status = 'processing') = (completed_at IS NULL)),
  CHECK (status <> 'succeeded' OR error_code IS NULL),
  CHECK (status <> 'failed' OR error_code IS NOT NULL)
);

ALTER TABLE conversation_messages
  ADD COLUMN turn_id UUID NULL REFERENCES conversation_turns(id) ON DELETE SET NULL;
ALTER TABLE conversation_tool_calls
  ADD COLUMN turn_id UUID NULL REFERENCES conversation_turns(id) ON DELETE SET NULL;
ALTER TABLE usage_events
  ADD COLUMN turn_id UUID NULL REFERENCES conversation_turns(id) ON DELETE SET NULL;
```

Indexes cover `(conversation_id, started_at)`, message `turn_id`, tool-call `turn_id`, and usage
`turn_id`. Conversations enforce `status=deleted` exactly when `deleted_at` is non-null; an
`anonymized_at` value is permitted only for a deleted conversation.

`completeTurn` requires the owned active conversation, the server-issued processing turn ID, status
`succeeded|failed`, non-negative integer latency, ordered messages, citations, tool calls, safe usage,
and optional summary/title updates. One database transaction:

- locks and verifies the conversation and processing turn;
- inserts ordered messages linked to the turn;
- inserts bounded tool-call audit rows linked to the turn and optional message;
- inserts one `usage_events.event_type=chat_turn` row linked to conversation and turn;
- updates the bounded summary/title and conversation timestamp;
- marks the turn succeeded or failed with completion time, latency, and safe error code.

Any failure rolls back the complete audit transition, leaving the turn `processing` for deterministic
recovery. A later retry may complete the same processing turn once; an already completed turn returns
its stored result without duplicating child rows when the requested terminal status matches, and
otherwise conflicts.

Messages are bounded to 100 per turn and 20,000 UTF-16 code units each. Tool calls are bounded to 50
per turn; tool name/correlation/error strings use stable server-owned grammars. Citations, message
metadata, tool arguments/results, turn metadata, and checkpoint JSON each serialize to at most
`CONVERSATION_AUDIT_JSON_BYTES`, default `65536`, accepted range `1024..1048576`. Token/retrieval
counts and latency are non-negative safe integers.

Accepted audit objects are strict. The recursively prohibited keys `provider`, `provider_name`,
`provider_id`, `provider_response_id`, `provider_conversation_id`, `response_id`, and `model` are
rejected from conversation, turn, message, tool, citation, usage, and checkpoint metadata. Provider
identity may exist only in ephemeral logs/metrics; application tables contain no provider
discriminator or provider-hosted state.

## 5. PostgreSQL Checkpoint Store

SV-US-010 uses the existing `langgraph_checkpoints` table as a local PostgreSQL checkpoint store. It
exposes `put`, `get`, and `getLatest` application methods compatible with the later graph adapter:

- `put` accepts only the server-owned conversation thread ID, non-empty checkpoint ID and optional
  parent ID (each at most 200 characters), bounded JSON checkpoint, and bounded safe metadata;
- a new `(thread_id, checkpoint_id)` is immutable;
- retrying the exact same checkpoint is idempotent;
- reusing an ID with different parent, checkpoint, or metadata returns `checkpoint_conflict`;
- `get` and `getLatest` return parsed JSON plus IDs and creation time; latest ordering is
  `created_at DESC, checkpoint_id DESC`;
- ownership-facing services resolve the thread ID through the conversation row; callers cannot read
  or write checkpoints by arbitrary thread ID.

Checkpoint writes use their own short transaction so graph progress can survive a process restart.
Completing a turn does not erase checkpoints. Soft deletion/anonymization deletes every checkpoint
for the conversation thread immediately.

## 6. Authenticated History Route

`GET /conversations/:id` requires an active installation credential with `conversations:read` and
exactly one visitor identity header:

- `X-Ask-Sunny-External-User-ID`, or
- `X-Ask-Sunny-Anonymous-Session-ID`.

Identity values follow section 2. The route returns the owned active/archived conversation ID, title,
status, summary, creation/update times, bounded recent messages, and `history_truncated`. It never
returns the stored identity, graph thread ID, turn metadata, raw tool arguments/results, or deleted
conversation data. The same `conversation_not_found` envelope covers absence, mismatch, or deletion.

## 7. Deletion, Anonymization, And Retention

These are internal service/CLI boundaries in SV-US-010; SV-US-012 may expose protected operational
controls later.

`softDeleteOwned` verifies exact ownership and, in one transaction:

- sets `status=deleted`, `deleted_at=NOW()`, `anonymized_at=NOW()`, clears identity columns, title,
  summary, and conversation metadata;
- replaces message content with `[deleted]` and clears citations/metadata/token counts;
- clears tool arguments/results while retaining tool name, status, latency, and safe error code;
- clears turn metadata while retaining status, timing, correlation, and safe error code;
- nulls direct identity fields and clears metadata on related usage rows;
- deletes every checkpoint for the thread.

The result is idempotent. After the first scrub the caller can no longer prove ownership, so a repeat
through the public ownership boundary returns the same non-disclosing not-found response. Internal
retention operations identify already-deleted rows without restoring identity.

`anonymizeExternalUser` applies the same scrub to every conversation for one exact external user,
deletes that user's profile and favorites, and nulls its direct identifier on remaining usage rows.
It never affects anonymous or other external identities.

`CONVERSATION_RETENTION_DAYS` defaults to `90` and accepts `1..3650`.
`CONVERSATION_DELETED_GRACE_DAYS` defaults to `30` and accepts `1..365`. The retention sweep:

1. marks `processing` turns older than 15 minutes as failed with `error_code=turn_interrupted`;
2. applies the complete scrub above to active/archived conversations whose `updated_at` is older than
   the retention cutoff;
3. hard-deletes already scrubbed conversations whose `deleted_at` is older than the grace cutoff;
4. nulls remaining external/anonymous identifiers on usage rows older than the retention cutoff.

The sweep is bounded by an explicit batch limit (`1..1000`), uses row locking with `SKIP LOCKED`, and
returns only safe counts. It is safe to retry and never logs identities, messages, tool payloads, or
checkpoints.

## 8. Failure And Security Rules

- Invalid shape/identity: `422 conversation_request_invalid` for internal boundaries and the common
  HTTP validation envelope at the route boundary.
- Missing, mismatched, deleted, or disallowed status: `404 conversation_not_found`.
- Completed-turn mismatch or checkpoint ID reuse with different state: `409` conflict.
- Database unavailable: `503 conversation_unavailable`; do not invent empty history or successful
  persistence.
- No operation logs visitor identity, message content, summary, citations, tool arguments/results,
  checkpoint state, or provider data.

## 9. Verification

Completion requires unit and live PostgreSQL evidence for anonymous/logged creation, exact ownership,
bounded chronological history, summary loading, non-disclosing cross-identity/deleted behavior,
successful and failed audit transactions, idempotent completion, prohibited provider fields,
checkpoint conflict/latest/restart recovery, soft-deletion scrubbing, external-user anonymization,
interrupted-turn recovery, retention batching, and hard-deletion after grace.
